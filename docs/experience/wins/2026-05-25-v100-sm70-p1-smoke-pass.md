# V100 sm_70 P1.4 smoke pass

## Context

P1 build-pass proved that ARLE could emit sm_70 cubins. P1.4 closed the next
gate: `Qwen3.5-4B` on Tesla V100-SXM2-32GB can start `arle serve`, run the
Qwen3.5 hybrid path, and return tokens through the OpenAI `/v1/completions`
surface.

The important debugging lesson was the initial HTTP failure:

```text
Inference request aborted before completion (server overloaded or out of memory). Please retry.
```

That was a catch-all placeholder, not root-cause evidence. It collapsed kernel,
model, scheduler, and client-abort failures into a fake "overloaded/OOM" bucket.
P1.4 therefore fixed the error chain before accepting the smoke result.

## What Worked

- **GDR sm_70 scope expanded.** The build support commit first admitted sm_70
  dense attention cubins; P1.4 flipped Qwen3.5 GDR TileLang kernels to
  `allow_sm70=true` as well. That mattered because the Qwen3.5-4B runtime smoke
  exercises the hybrid/GDR path, not a dense-attention-only shortcut.
- **Error-chain hygiene is now usable.** HTTP streaming and non-streaming paths
  propagate `CompletionStreamError { kind, message, chain }`; scheduler aborts
  preserve `AbortReason`; Qwen3.5 prefill/decode calls add layer/op context; and
  Marlin sm_70 stubs normalize to `cudaErrorNotSupported` with kernel/shape
  params at the Rust FFI boundary.
- **Debug sync is gated.** `ARLE_CUDA_DEBUG_SYNC=1` can force CUDA sync probes
  during triage, but the default release path does not pay sync cost.
- **TileLang PR #2257 is upstream-ready for review.** The PR is now draft, with
  clang-format + ruff-format fixes pushed (`7e15984d` in the TileLang fork) and
  lint clean for the local formatting gates.

## Root Cause Notes

- The "server overloaded or out of memory" response was inaccurate. It was an
  HTTP catch-all for a scheduler channel closing without a terminal delta, not
  proof of VRAM OOM.
- After rebuilding the actual binary that `arle serve` launches
  (`target/release/infer`, not only `target/release/arle`) and enabling the GDR
  sm_70 cubins, the same V100 smoke returned tokens with no CUDA/OOM/error log
  markers.
- The concrete root cause class was therefore missing/incorrect sm_70 operator
  coverage plus stale binary verification, not a confirmed
  `CUDA_ERROR_OUT_OF_MEMORY`.

## Rule

For V100 follow-up, never treat the generic HTTP 503 text as evidence. A useful
failure must carry request lifecycle kind, model op/layer context, and CUDA /
kernel status. If P3/P4 fails, start from the structured error chain and the
server trace, not from cubin-inventory guesses.
