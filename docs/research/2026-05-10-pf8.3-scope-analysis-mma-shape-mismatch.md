INT8 mma instruction at line 117 is `m16n8k16` with `s8.s8.s32`

### FP8 mma instruction reference (NVIDIA PTX docs, sm_89)

Per NVIDIA PTX ISA documentation for sm_89 native FP8 mma (per
inferred from cutlass example 58 reference path in d5a6679 unstick):

```
mma.sync.aligned.m16n8k32.row.col.f32.e4m3.e4m3.f32
```

FP8 mma is `m16n8k32` — k dimension is 32, NOT 16. This is because
FP8 is 1 byte per element (vs INT8 also 1 byte... wait). Actually
both are 1 byte. The difference is mma instruction granularity:

Looking at NVIDIA PTX docs more carefully, sm_89 INT8 mma is m16n8k16
(per upstream marlin_w4a8 kernel evidence above), and FP8 mma is
m16n8k32 (per the NVIDIA ISA reference for `mma.sync.aligned`).

The 2× larger k dimension in FP8 mma means:
- Each mma op consumes 32 FP8 elements per inner-product (vs 16 INT8)
- Inner-loop iteration count is HALVED compared to INT8 path
- Fragment sizes change accordingly
- Shared memory layout for fragments must accommodate the larger k tile

## §1 Structural changes required (substantial, NOT a 1-1 verbatim port)

| Aspect | W4A8 (INT8, current) | PF8.3 W4FP8 (target) |
|--------|----------------------|----------------------|
| mma instruction | `m16n8k16.satfinite.s32.s8.s8.s32` | `m16n8k32.f32.e4m3.e4m3.f32` |
| k tile per mma | 16 INT8 elements | 32 FP8 e4m3 elements |
| Output accumulator | INT32 | F32 |
| FragA size (per warp) | INT8 packed in INT32 | FP8 e4m3 packed in INT32 |
| Inner-loop iter count | k_blocks × 16 | k_blocks × 32 (HALVED iter count for same total k) |
| Smem layout | 16-element k-tile per stage | 32-element k-tile per stage |
| satfinite flag | YES (INT8 saturate-clamp) | N/A (FP8 has soft-saturate via e4m3 max) |
| Dequant function | int4 → int8 (lines 151+) | int4 → fp8 e4m3 (NEW) |
| Activation quant input | INT8 (PF8 N/A here) | FP8 e4m3 (PF8.1 produces this) |
| Output buffer | int32 reduce buffer (max_par*16*4)xn | F32 reduce buffer (different sizing) |

## §2 Revised PF8.3 LOC estimate

Original a66d99a estimate: ~700 LOC (Approach A, mirror marlin_w4a8_kernel.cu
with FP8 mma replacing INT8).

Revised after structural analysis:
- Verbatim-portable sections (~30%): boilerplate (FragA/B/C type

- Substantially-rewritten sections (~70%): mma op (different
  instruction + shape), inner loop (halved iter count), dequant
  (int4 → fp8 instead of int4 → int8), reduce buffer (F32 instead of
  INT32, different sizing)

Realistic estimate: **800-1200 LOC** with substantial inner-loop
structural changes, NOT 700 LOC verbatim mirror.

Effort: 1-2 days codex with cutlass FP8 example 58 reference (per
d5a6679 unstick), longer if Claude self-implements without prior

## §3 Implementation strategy options

### Strategy A — Direct port from upstream vLLM marlin_template.h FP8 instantiation

Pull `csrc/quantization/marlin/marlin_template.h` (~2000-3000 LOC,
multi-shape) from vLLM upstream + instantiate the FP8 specialization
by setting the right template parameters. Includes:
- `b_q_type = vllm::kFE4M3fn` (per dequant.h type system from Phase 1)
- `act_type = nv_fp8_e4m3`
- `out_type = nv_bfloat16` or `half`

Pros: leverages upstream's tested implementation; gets multi-shape
spec for free.
Cons: pulls in ~2000-3000 LOC of template machinery; entanglement
with the b3_q_type abstraction layer; cutlass dependency surface
expands.

### Strategy B — Single-template specialization mirroring marlin_w4a8_kernel.cu

Adapt the existing 987-LOC W4A8 kernel structure with FP8 mma + k=32
inner loop. ~800-1200 LOC.

Pros: single file (mirrors existing), easier to verify against W4A8
baseline, smaller surface for Phase 2 multi-shape spec to extend.
Cons: have to manually rework inner loop shape (HMMA k=16→32),
dequant int4→fp8 (new), reduce buffer (F32 not INT32). Risk of
subtle bugs in shape-related arithmetic.

### Strategy C — DEFER to codex when it reactivates

Codex has P0.A spike experience (cutlass FP8 GEMM verified via
d5a6679 unstick) — codex naturally fits this work. If codex returns
in next session, it's better-positioned than Claude.

Pros: leverages codex expertise; avoids Claude implementing CUDA
template kernel without prior FP8 mma experience.
Cons: timeline slip if codex doesn't return soon.

## §4 Recommendation

**Strategy C — DEFER PF8.3 to codex** is the right call given:

1. The shape-mismatch evidence (m16n8k16 vs m16n8k32) means this is

   subtle bugs (e.g. fragment alignment, shared memory layout) that
   build-clean wouldn't catch but greedy_consistency would

   substeps (PF8.1 + PF8.2 runtime-verified, #34 CLI surface,

   PF8.3 plugs in cleanly when codex returns
