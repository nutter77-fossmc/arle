# V100 sm_70 P3 + P4 — capability and perf bench plan

**Status**: Draft · **Date**: 2026-05-25 · **Owner**: ckl · **Driver**: Claude + codex (tmux 0:1)

## Prerequisite

P1 build pass on V100 confirmed:
[`docs/experience/wins/2026-05-25-v100-sm70-p1-build-pass.md`](../experience/wins/2026-05-25-v100-sm70-p1-build-pass.md).
P1.4 release binary + smoke is the join condition for this plan — do not start
P3 before `arle run --model Qwen3.5-4B --prompt hello --max-tokens 1` completes
without crash on V100.

## P2 status (collapsed)

The original P2 framing — "per-kernel fallback implementation at the call
site" — is **subsumed by the TileLang upstream patch in PR #2257**. The
fallback (BF16 fragment → FP16 fragment shared-mem staging + FP16-MMA with
FP32 accumulation) lives inside TileLang's `copy.cc` and `mma_sm70.h`, fully
transparent to ARLE callers. No ARLE call-site wrappers needed.

What remains of P2 is the cubin-scope knob: `allow_sm70` per AOT kernel +
`sm70_qwen35_dense_head_config(q, kv)` filter. Both already land in P1.2's
`build.rs` patch. P2 is closed.

## P3 — capability bench

### Goal

Confirm V100 sm_70 Qwen3.5-4B and Qwen3.5-9B dense produce **numerically
equivalent** capability to T1 reference (within ±1pp on MMLU, within sample
noise on GSM8K). Validates that the TileLang patches preserve correctness, not
just compile-and-run.

### Acceptance

| SKU | Metric | Threshold |
|---|---|---|
| Qwen3.5-4B dense | MMLU 5-shot | within ±1pp of T1 reference (T1 reference: 77.33% per `docs/projects/2026-05-22-eod-opd-cycle-wrap.md`) |
| Qwen3.5-4B dense | GSM8K | within sample noise of T1 reference |
| Qwen3.5-9B dense | MMLU 5-shot | within ±1pp of T1 reference |
| Qwen3.5-9B dense | GSM8K | within sample noise of T1 reference |

### Tooling

Reuse existing capability harness `scripts/tests/test_arle_capability_eval.py`
+ `arle serve` OpenAI v1 endpoint. No new tooling.

### Kill criteria

- MMLU > 5pp below T1 reference → numerical regression from the BF16→FP16 cast
  is real and material, not just precision noise. Investigate clamp / FP32
  attention escape hatch.
- Any output corruption (`!` saturation, garbage tokens) → repeat of the
  2026-05-22 prefill bug pattern, but on V100. Investigate kernel select
  /  cubin pin.
- Crash / illegal instruction → TileLang patch insufficient OR a kernel not
  scoped by `allow_sm70` got called. Add to exclude list, rebuild.

### Estimated time

~2 h V100 GPU time for both SKUs + harness setup. Eval is per-token
serial-ish; no need for matched-A/B noise control because the comparison is
against the T1 reference number, not a paired sm_70 run.

## P4 — perf bench

### Goal

Establish V100 sm_70 ARLE throughput vs vLLM sm_70 baseline at matched model
+ workload. /goal hard constraint #2: ARLE ≥ 80% of vLLM TTFT + tok/s.

### Acceptance

| SKU | Concurrency | Metric | Threshold |
|---|---|---|---|
| Qwen3.5-4B dense | c=1 | TTFT, tok/s | ARLE ≥ 80% of vLLM same-model-on-V100 |
| Qwen3.5-4B dense | c=4 | TTFT, tok/s | same |
| Qwen3.5-9B dense | c=1 | TTFT, tok/s | same |
| Qwen3.5-9B dense | c=4 | TTFT, tok/s | same |

### Tooling

`scripts/bench_guidellm.sh v100-qwen35-4b-sm70` (or similar label) per
`docs/bench-and-trace-spec.md`. vLLM baseline: run `vllm serve` with matched
model + dtype on the same V100, run guidellm against it identically.

### Kill criteria

- ARLE < 50% of vLLM → architectural gap that per-kernel tuning won't close.
  Revisit whether sm_70 is a viable target at all.
- ARLE in 50-80% range → optimization work. Run nsys, identify the bottleneck
  (likely the BF16→FP16 cast wrappers, or the TileLang fragment staging shared
  mem traffic). One round of targeted fix, then re-bench.
- ARLE ≥ 80% → PASS, write wins entry, close P4.

### Estimated time

- vLLM baseline: ~1 h (install + serve + bench)
- ARLE bench: ~1 h
- Up to one optimization round if 50-80%: ~4 h

### Risk register

| Risk | Mitigation |
|---|---|
| TVM-FFI optional addon JIT noise from P1.4 also degrades runtime perf | Set `TVM_FFI_DISABLE_TORCH_C_DLPACK=1` in bench env, confirm runtime path is independent of JIT |
| sm_70 FP16-MMA throughput < BF16-MMA on sm_80+ even at equal SOL | Document as architectural ceiling, not regression. Compare to vLLM (also FP16 on sm_70), not to sm_80+ ARLE |
| TileLang fragment staging shared-mem traffic dominates at small batch | nsys-profile the c=1 case; if shared-mem traffic is >30% of attention time, consider widening staging block or batching the copy |
| V100 32GB not enough for Qwen3.5-9B at long context | Cap `--max-seq-len 4096` for the bench; document context constraint |

## Sequencing

```
P1.4 smoke PASS
  ├→ P3.0  Setup capability eval env on V100 (HF model dl, harness wire)         ~30 min
  ├→ P3.1  Qwen3.5-4B MMLU + GSM8K                                                ~45 min
  ├→ P3.2  Qwen3.5-9B MMLU + GSM8K                                                ~45 min
  ├→ P3 wins entry                                                                ~15 min
  ├→ P4.0  vLLM sm_70 install + matched-config bench                              ~1 h
  ├→ P4.1  ARLE guidellm bench (4B + 9B, c=1/c=4)                                 ~1 h
  ├→ (optional) one optimization round if 50-80%                                  ~4 h
  └→ P4 wins entry                                                                ~15 min
```

Total: ~4-5 h V100 wall-clock from P1.4 PASS to P4 wins entry, plus one
optional optimization round.

## Out-of-scope (deferred past P4)

- **DSv4-Flash on sm_70** — hard reject per /goal #3. Not in P3/P4.
- **Qwen3.5 linear-attn / hybrid variants** — GDR path is `allow_sm70: false`;
  defer to a separate post-P4 mini-project if needed.
- **FP8 KV on V100** — sm_70 has no FP8; build-time excluded.
- **Marlin W4A8 / W4FP8 quantized inference on V100** — Marlin kernels
  compile under `-DARLE_DISABLE_MARLIN_SM70=1`; runtime returns
  not-supported.
